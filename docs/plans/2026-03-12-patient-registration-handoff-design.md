# 患者注册后业务承接设计

## 背景

当前患者自助注册已经具备真实后端接口，`/api/auth/register` 可以创建患者账号，随后也能通过 `/api/auth/login` 正常登录。后端 `AuthIntegrationTest` 已经证明了这条链路是可用的。

问题出在前端承接层：登录页注册成功后，只会提示“请使用新账号登录”，不会自动建立会话，也不会把新患者带进患者工作台。对演示和真实使用来说，这会让“注册完成”与“开始使用系统”之间出现明显断点。

本轮目标是补齐这个最小承接闭环，让新患者注册成功后能立即进入患者端首页，并在首次进入时得到明确但轻量的下一步引导。

## 目标

- 患者注册成功后自动登录
- 自动登录成功后直接进入患者首页 `/patient/dashboard`
- 患者首页在“刚注册首次进入”时展示轻量引导卡
- 引导卡明确告诉用户下一步去预约挂号
- 自动登录失败时保留注册成果，并回退到可继续手动登录的状态
- 不改后端注册协议，不新增 token 返回

## 非目标

- 不新增注册后专用欢迎页
- 不改后端 `/api/auth/register` 响应结构
- 不新增预约页、缴费页的新手流程
- 不引入复杂的 onboarding 多步骤引导
- 不改患者首页已有真实数据加载方式

## 方案选择

本轮采用“自动登录 + 轻引导首页版”：

- 注册成功后前端立刻复用现有 `login()` 能力建立会话
- 成功后跳转到患者首页，而不是停留在登录页
- 用一个只生效一次的轻量引导标记驱动首页顶部卡片
- 保持后端接口和认证协议不变，尽量把改动压在前端承接层

相比“只自动登录不做首页引导”，这版能更好地解释新患者下一步该做什么；相比“直接跳预约页”，这版更符合患者首页作为工作台总览入口的定位。

## 会话承接设计

### 登录页注册成功后的动作

在 `LoginPage.vue` 中：

1. 调用 `registerPatient()` 提交注册
2. 注册成功后，立刻调用现有 `login()`，使用刚创建的用户名和密码建立会话
3. 自动登录成功后，写入一个“首次注册引导”标记
4. 通过路由跳转到 `roleMeta.patient.homePath`，即 `/patient/dashboard`

这样可以复用现有登录能力，避免额外引入新的认证分支。

### 自动登录失败后的回退

如果注册成功但自动登录失败：

- 保留注册成功结果，不进行任何回滚
- 把刚注册的用户名和密码回填到登录表单
- 关闭注册弹窗
- 提示“账号已创建，请手动登录”

这样用户仍然能在当前页马上完成下一步，不会因为自动登录失败而失去注册成果。

## 首次进入首页引导设计

### 引导标记

在 `patientSession.js` 增加轻量的注册引导状态管理，基于 `sessionStorage` 保存一次性数据，例如：

- `displayName`
- `patientId`
- `username`
- `createdAt` 或简单布尔值

该状态只服务于“当前浏览器会话里的首次进入首页”，不需要长期持久化到 `localStorage`。

### 首页展示方式

在 `PatientDashboard.vue` 顶部新增一张轻量引导卡，仅在检测到注册引导标记时显示。卡片内容包括：

- 欢迎语
- 患者编号
- “你已完成注册，下一步建议先预约挂号”的说明
- 主按钮：`去预约挂号`
- 次按钮：`稍后再说` 或简单关闭入口

引导卡关闭后要消费掉该标记，避免后续普通登录重复出现。

### 与已有空状态的关系

患者首页原有的统计卡、时间线和快捷入口保持不变。引导卡只是额外放在顶部，帮助新患者理解第一步操作，不替代现有真实数据空状态。

如果新患者没有任何预约记录：

- 统计卡继续显示 0
- 时间线继续显示空状态
- 引导卡负责把“下一步去预约挂号”说清楚

## 前端改动范围

### `frontend/src/views/common/LoginPage.vue`

- 注册成功后改为自动登录
- 自动登录成功后跳转患者首页
- 自动登录失败时回填登录表单并提示手动登录
- 优化注册成功提示文案，使其从“请使用新账号登录”变为“已自动进入患者端”或“账号已创建，请手动登录”

### `frontend/src/services/patientSession.js`

新增最小 onboarding 状态接口，例如：

- `setRegistrationOnboarding(data)`
- `getRegistrationOnboarding()`
- `clearRegistrationOnboarding()`

并保持 `getActivePatient()` 逻辑不被破坏。

### `frontend/src/views/patient/PatientDashboard.vue`

- 检查是否存在注册 onboarding
- 在页面顶部显示一次性欢迎引导卡
- 提供跳转到 `/patient/appointments` 的主操作
- 在用户关闭或消费引导后清除该状态

### `frontend/src/services/patientDashboard.js`

如有必要，将首页引导卡内容抽成简单的模型构建函数，方便脚本测试；但不强行过度抽象。

## 错误边界

- 注册请求失败：沿用现有错误提示
- 注册成功但自动登录失败：不丢数据，回填登录表单并提示继续登录
- sessionStorage 不可用时：不阻断登录，仅退化为没有首次引导卡
- 患者首页真实数据加载失败时：沿用现有错误提示和空模型，不影响首次欢迎卡展示

## 测试设计

### 后端

本轮不改后端注册协议，但仍建议跑全量 `mvn test`，确认前端承接变更没有依赖错误假设。

### 前端

扩展或新增脚本覆盖以下行为：

- 登录页存在“注册成功后自动登录”的关键调用链路
- 自动登录失败回退逻辑存在
- `patientSession.js` 提供注册 onboarding 状态接口
- 患者首页存在首次欢迎卡与“去预约挂号”主按钮
- `patientDashboard.js` 如新增了引导模型，也要覆盖其空状态与文案

### 全量验证

- `backend`: `mvn test`
- `frontend`: 新的注册承接脚本
- `frontend`: `node ./scripts/test-patient-dashboard.mjs`
- `frontend`: `npm run build`
- `frontend`: `npm run verify:bundle`

## 风险与控制

- 自动登录属于注册后的新增动作，风险集中在前端流程控制；通过复用现有 `login()` 降低实现风险
- 首次引导如果持久化过重，容易影响后续普通登录；因此只采用一次性 `sessionStorage` 标记
- 新增引导卡不应挤占首页主体信息，因此只做轻量卡片，不做全屏欢迎页
