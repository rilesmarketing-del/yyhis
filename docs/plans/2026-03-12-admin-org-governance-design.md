# 管理端组织与账号治理设计

## 背景

当前管理端组织页已经支持科室树查看、角色统计、账号新增和科室新增，但还缺少真正的治理闭环。管理员无法编辑已有账号，无法启停账号，无法重置密码，也无法调整科室结构。这会让演示停留在“能建数据”而不是“能管理数据”的阶段。

本轮目标是把组织页升级成最小可用的治理页，同时保持当前系统是演示级实现，不引入删除、批量处理和复杂审批。

## 目标

- 支持编辑科室名称与上级科室
- 支持编辑账号资料
- 支持启用和停用账号
- 支持将账号密码重置为统一默认密码 `123456`
- 禁止管理员停用自己
- 保持现有树 + 表布局，不重做页面架构

## 非目标

- 不做账号删除
- 不做科室删除
- 不做批量启停
- 不做密码强度、密码历史、首次登录改密等安全增强
- 不改用户名

## 方案选择

本轮采用“平衡闭环版”：

- 后端补最小治理接口，复用现有 `UserAccount`、`Department`、`AdminOrgSummaryResponse`
- 前端保留现有页面结构，只增加治理动作和编辑弹窗
- 变更成功后统一刷新汇总接口，不做局部状态拼补

这样可以在尽量小的改动下，把新增/查看页补成真正可管理的治理页。

## 后端设计

### 接口

在 `AdminOrgController` 增加以下接口：

- `PUT /api/admin/departments/{departmentId}`
- `PUT /api/admin/accounts/{username}`
- `POST /api/admin/accounts/{username}/enable`
- `POST /api/admin/accounts/{username}/disable`
- `POST /api/admin/accounts/{username}/reset-password`

### 科室编辑

后端支持修改：

- 科室名称
- 上级科室

约束：

- 名称不能为空
- 同级科室名称不能重复
- 新上级必须存在
- 科室不能挂到自己名下
- 科室不能挂到自己的任意子节点下，避免形成环

### 账号治理

后端支持修改：

- `displayName`
- `role`
- `departmentId`
- `title`
- `mobile`

约束：

- 用户名不可改
- 医生和管理员必须绑定科室
- 患者可以不绑定科室
- 角色改成患者时，清空科室与职称
- 角色改成医生或管理员时，必须重新校验科室合法性

### 启停规则

- 只改 `enabled`
- 不删除历史预约、排班、接诊、病历等记录
- 停用自己直接拒绝
- 启停成功后，汇总页继续通过 `enabled` 展示状态

### 密码重置

- 统一重置为 `123456`
- 不需要传新密码
- 成功后只返回更新后的账号信息

## 前端设计

### 页面结构

保留当前：

- 左侧科室树 + 角色统计
- 右侧人员表

新增：

- 科室树节点“编辑科室”入口
- 人员表“编辑 / 启用或停用 / 重置密码”操作列

### 弹窗复用

- 科室弹窗同时支持新增和编辑
- 账号弹窗同时支持新增和编辑
- 编辑账号时不展示密码输入框
- 新增账号时继续填写初始密码

### 风险提示

- 停用账号需要确认
- 重置密码需要确认，并提示“将重置为 123456”
- 当前登录管理员自己的停用按钮禁用

### 刷新策略

每次治理动作成功后都重新拉取 `/api/admin/org/summary`，保证树、表、角色统计三块数据同步。

## 测试设计

### 后端

扩展 `AdminOrgIntegrationTest`，覆盖：

- 科室改名与调整上级
- 账号编辑
- 账号停用后无法登录
- 账号重新启用后可再次登录
- 密码重置为 `123456`
- 管理员不能停用自己

### 前端

新增组织治理脚本，覆盖：

- `adminOrg.js` 映射逻辑
- 页面关键中文文案
- 编辑、停用、重置密码等操作入口是否存在

### 全量验证

- `backend`: `mvn test`
- `frontend`: 新组织治理脚本
- `frontend`: `npm run build`
- `frontend`: `npm run verify:bundle`

## 风险与控制

- 角色切换可能影响既有业务绑定，本轮只允许修改资料字段，不改用户名，减少联动风险
- 科室树调整容易引入循环引用，后端强校验并在前端过滤非法上级选项
- 停用自己会破坏演示入口，前后端双层拦截
