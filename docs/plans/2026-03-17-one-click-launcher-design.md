# 智慧医院系统一键启动器设计

## 背景

当前项目已经具备本地手动启动前后端的能力，但日常测试需要分别启动后端和前端，再手动打开浏览器，步骤偏多，也容易因为忘记某一步导致误判系统状态。

这次目标是提供一个 Windows 双击即用的一键启动入口，让使用者不需要打开终端，就能完成：

- 检查前后端是否已运行
- 启动缺失的服务
- 等待前端就绪
- 自动打开系统首页

## 目标

- 在项目根目录提供一个可双击的启动程序，命名为 `智慧医院系统.cmd`
- 启动程序默认静默运行，不弹出长期驻留的终端窗口
- 如果前后端已在运行，不重复启动，直接打开首页
- 如果只有一侧未运行，只补启动缺失的服务
- 浏览器自动打开 `http://127.0.0.1:5173`
- 运行日志可落盘，方便排障

## 非目标

- 不做系统托盘程序
- 不做安装包或 `.exe`
- 不增加停止服务脚本
- 不额外支持自定义端口、环境切换或多实例运行

## 方案选择

### 方案一：单个 `.cmd` 直接写完全部逻辑

优点：

- 文件数量最少
- 用户看起来最直接

缺点：

- 端口探测、等待轮询、日志重定向在批处理里可读性差
- 后续维护成本高

### 方案二：`.cmd` 作为入口，PowerShell 承担主逻辑

优点：

- 用户仍然是双击一个文件
- PowerShell 更适合做端口检查、后台启动、轮询等待和错误输出
- 后续扩展更稳

缺点：

- 文件会多一个

### 方案三：只提供 `.ps1`

优点：

- 实现最干净

缺点：

- 日常使用门槛更高
- 不符合“双击就能用”的目标

## 结论

采用方案二：

- 根目录新增 `智慧医院系统.cmd`
- 根目录新增 `launch-hospital-system.ps1`
- `.cmd` 只负责转发到 PowerShell
- PowerShell 负责服务检查、启动、等待、日志和打开浏览器

## 运行流程

### 正常流程

1. 用户双击 `智慧医院系统.cmd`
2. 脚本进入 `launch-hospital-system.ps1`
3. 检查 `5173` 和 `8080` 是否已有监听
4. 对未运行的服务执行后台启动
5. 轮询等待前端首页可访问
6. 打开默认浏览器进入 `http://127.0.0.1:5173`

### 已运行场景

如果前后端都已运行：

- 不重复拉起进程
- 直接打开首页

如果只运行了一侧：

- 只启动缺失的一侧

## 启动命令

### 后端

- 工作目录：`backend`
- 启动命令：`mvn spring-boot:run`
- 日志输出：
  - `backend/launch-backend.log`
  - `backend/launch-backend.err.log`

### 前端

- 工作目录：`frontend`
- 启动命令：`npm run dev -- --host 127.0.0.1 --port 5173`
- 日志输出：
  - `frontend/launch-frontend.log`
  - `frontend/launch-frontend.err.log`

## 就绪判定

为了避免浏览器过早打开，前端就绪以 `http://127.0.0.1:5173` 返回成功响应为准。

脚本会：

- 先等待端口监听
- 再轮询 HTTP 首页
- 设定合理超时时间

若超时：

- 不打开浏览器
- 在命令行短暂输出“启动超时，请检查日志”

## 错误处理

- 如果 Maven 或 npm 命令不可用，脚本直接报错并退出
- 如果前端或后端启动失败，保留日志文件供排查
- 如果浏览器打开失败，不影响服务继续运行
- 如果当前目录被移动或脚本不在项目根目录，使用脚本自身路径推导项目根目录
